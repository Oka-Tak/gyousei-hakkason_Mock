# PostgreSQL データインポート方法と理由に関する報告書

## 1. 概要

この報告書は、`project_spending_block` および `project_spending` テーブルへのデータインポート方法、および特に `block_id` の扱いに特殊な方法が必要となる理由について説明します。

## 2. インポートの全体フロー

PostgreSQLへのデータインポートは、以下の2段階のプロセスで実行されます。

1.  **`project_spending_block` テーブルへのインポート**:
    *   `converted_data/converted_project_spending_block.csv` ファイルを使用して、`project_spending_block` テーブルにデータをインポートします。
2.  **`project_spending` テーブルへのインポート**:
    *   `converted_data/converted_project_spending.csv` ファイルのデータと、既に `project_spending_block` テーブルにインポートされたデータから取得した `block_id` を組み合わせて、`project_spending` テーブルにデータをインポートします。

## 3. `project_spending_block` テーブルへのインポート方法と理由

### 方法

`project_spending_block` テーブルへのデータインポートには、PostgreSQLの `\COPY` コマンドを使用します。

```sql
\COPY project_spending_block (project_id, budget_year, block_no, block_name, role_in_project, block_total_amount) FROM 'converted_data/converted_project_spending_block.csv' WITH (FORMAT CSV, HEADER TRUE, ENCODING 'UTF8');
```

### 理由

この方法が採用される主な理由は、`project_spending_block` テーブルの `block_id` 列が `BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY` として定義されているためです。

*   **`GENERATED BY DEFAULT AS IDENTITY`**: この定義は、データベースが新しいレコードが挿入される際に、`block_id` の値を自動的に生成することを意味します。CSVファイルに `block_id` を含めてしまうと、データベースの自動生成メカニズムと競合し、エラーが発生する可能性があります。
*   **主キーの自動生成**: `block_id` はテーブルの主キーであり、各レコードを一意に識別するためにデータベースが責任を持って管理すべきです。外部から提供される値ではなく、データベース内部で一貫性を持って生成されることが望ましいです。

したがって、CSVファイルには `block_id` を含めず、データベースにその生成を任せるのが正しいアプローチです。

## 4. `project_spending` テーブルへのインポート方法と理由

### 方法

`project_spending` テーブルへのデータインポートは、直接 `\COPY` コマンドを使用するのではなく、Pythonスクリプト (`import_project_spending.py`) を介して行われます。

```python
# import_project_spending.py (抜粋)
# ...
cur.execute(
    "SELECT block_id FROM project_spending_block WHERE project_id = %s AND budget_year = %s AND block_no = %s",
    (project_id, budget_year, block_no)
)
block_id_result = cur.fetchone()

if not block_id_result:
    print(f"Skipping row: No matching block_id found for project_id={project_id}, budget_year={budget_year}, block_no='{block_no}'")
    continue

block_id = block_id_result[0]

# ... その他のデータを準備し、block_id を含めて INSERT
```

### 理由

この特殊な方法が必要となる理由は、`project_spending` テーブルが `project_spending_block` テーブルの `block_id` を外部キーとして参照しているためです。

*   **外部キー制約**: `project_spending` テーブルの `block_id` 列は、`project_spending_block` テーブルの `block_id` 列に存在する値でなければなりません。
*   **自動生成された `block_id` の参照**: `project_spending_block` の `block_id` はデータベースによって自動生成されるため、CSVファイルにはその値が含まれていません。`project_spending` のCSVには、`project_spending_block` を一意に特定するための `project_id`, `budget_year`, `block_no` の組み合わせが含まれています。
*   **ルックアップの必要性**: `project_spending` のデータを挿入する前に、各レコードに対応する `project_id`, `budget_year`, `block_no` を使用して `project_spending_block` テーブルから実際に生成された `block_id` を検索（ルックアップ）し、その取得した `block_id` を `project_spending` テーブルの `INSERT` ステートメントに含める必要があります。

この2段階のインポートプロセスとPythonスクリプトによるルックアップは、データベースの自動生成される主キーと外部キーの整合性を保ちながら、関連するテーブルにデータを正確にインポートするために不可欠です。
